## Working Methodology

### When designing or creating new content/pages
1. Use **project-analizer** skill to understand the current project structure and conventions
2. Follow the existing sidebar, naming, and MDX patterns already established in the project
3. Create files using `UPPER_SNAKE_CASE.md` for guideline docs, or the appropriate Docusaurus naming for pages and components
4. Always include valid frontmatter (`title`, `description`, `sidebar_position` where applicable)
5. Validate new pages render cleanly: run `npm run build` or `bun run build` and check for errors

### When editing existing content
1. Read the file first — understand its structure, frontmatter, and MDX usage before modifying
2. Preserve existing formatting conventions, heading hierarchy, and code block language tags
3. Run typecheck (`npm run typecheck` or `bun run typecheck`) after any TypeScript changes
4. Run build to confirm no broken imports, missing assets, or MDX errors

### When troubleshooting
1. Read the error output carefully — Docusaurus errors are usually precise about file and line
2. For MDX parse errors: check unclosed tags, invalid JSX, or unsupported syntax
3. For plugin errors: verify the plugin is installed (`package.json`) and configured correctly in `docusaurus.config`
4. For Vercel build failures: use **web-search** skill to look up the specific error if unfamiliar
5. For Node/dependency version issues: check `.nvmrc`, `engines` field in `package.json`, and Vercel's Node version setting

### When deploying to Vercel
1. Confirm the build succeeds locally before pushing: `npm run build` or `bun run build`
2. Verify `vercel.json` exists and is correctly configured if custom routing is needed
3. Use `vercel --prod` for production deployments via CLI, or push to `main` for automatic deployment
4. After deployment, verify the live URL renders correctly and all routes resolve

## Best Practices

- **Never commit secrets or API keys** — use environment variables and `.env.local` (gitignored)
- **Keep `docusaurus.config` typed** — use `docusaurus.config.ts` with TypeScript for autocomplete and safety
- **MDX components must be typed** — define prop interfaces in `PascalCaseProps` style
- **Sidebar should be explicit for guideline repos** — avoid fully auto-generated sidebars where document order matters
- **All docs must end with `## References` and `*Last Updated: DD Mon YYYY*`** per project conventions
- **Doc file names**: `UPPER_SNAKE_CASE.md` for guideline docs; follow Docusaurus conventions for blog posts and pages
- **Git workflow**: always use feature branches; never commit directly to `main`
- **Commit messages**: follow Conventional Commits (`docs:`, `feat:`, `fix:`, `chore:`); no AI attribution
- **Accessibility**: every page must be navigable by keyboard and screen reader; use semantic HTML in custom components
- **Performance**: avoid large client-side bundles in docs; prefer static rendering; lazy-load images
